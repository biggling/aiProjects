## 1. IRS 1099-DA Demand Signal (US Market)

### Current State of 1099-DA in 2026

The IRS Form 1099-DA rollout is live but deliberately incomplete — and that gap is the pain point.

**What is happening now:**
- For tax year 2025 (forms issued by February 17, 2026), brokers report **gross proceeds only**. Cost basis is intentionally NOT required until the 2026 tax year (forms arriving in early 2027).
- April 15, 2026 was the main federal filing deadline. US crypto investors had to reconcile their own cost basis against exchange-issued 1099-DAs that showed gross proceeds with no basis — a mismatch the IRS can see before taxpayers can explain it.
- IRS transition relief: no penalties for 2025 reporting if brokers made a good-faith effort. But the IRS still receives the gross proceeds data, and automated letters to reconcile discrepancies are expected.

**The gross proceeds problem:** The IRS sees `$4,250 in proceeds` but `$0 in cost basis`, implying a $4,250 taxable gain — when the real gain might be $2,450. Taxpayers who don't actively track cost basis are exposed to overpayment (best case) or audit letters (likely case).

**Multi-exchange problem:** If a user bought BTC on Coinbase and transferred to Kraken before selling, Kraken's 1099-DA shows the sale with no basis — because Kraken didn't see the original purchase. The user must self-document the transfer chain. This is exactly the reconstruction workflow an MCP can automate.

**Phase 2 begins January 1, 2026:** From this point forward, brokers must also report cost basis for "covered securities" (crypto purchased on their platform on or after Jan 1, 2026). The first forms with basis data arrive early 2027. This means 2026 is a **transition year** where user self-tracking is still mandatory for pre-2026 holdings.

### What Users Are Complaining About

Pain points documented in CPAs, audit firms, and tax guides in April 2026:

1. **Missing cost basis on 1099-DA** — exchanges only report proceeds; users must reconstruct basis themselves for any pre-2026 purchases.
2. **Cross-exchange transfers** — assets moved between wallets/exchanges break the cost basis chain; each exchange only sees its own leg.
3. **Staking, DeFi, airdrops** — not reported by any broker yet; fully the user's responsibility to value and declare.
4. **Forensic reconciliation burden** — accountants spending hours per client manually matching proceeds to client records.
5. **Zero-proceeds rejections** — some 1099-DAs filed with $0 proceeds are being rejected by the IRS, leaving brokers scrambling.
6. **Small gains / dust transactions** — many small DeFi interactions require individual cost basis per transaction; no automation exists in mainstream tools for this.

### KV vs D1 for MCP Data

**KV (Key-Value) is sufficient for:**
- User session tokens and API keys (encrypted at rest)
- Price cache (TTL 60s per token)
- User tier/subscription status

**KV falls short for:**
- Multi-wallet transaction history (structured query needs)
- Cost basis calculations that require joins across transactions
- Searching transaction history by date range, token, or exchange

**D1 (SQLite) is required for:**
- Storing imported transaction history per user
- Running FIFO/LIFO cost basis queries (require ORDER BY, WHERE, SUM aggregations)
- Per-user portfolio snapshots

**Verdict:** Use KV for API keys and cache. Use D1 for transaction storage. Both are available on the free tier and the $5/month paid plan. D1 at 5 million rows/day free is more than sufficient — a user with 10,000 transactions triggers maybe 50,000 row reads for a full cost basis calculation.

### MCP Streamable HTTP on Cloudflare Workers — Known Issues

- **Streamable HTTP is officially supported and recommended** by Cloudflare for new MCP servers. Cloudflare's own documentation uses `createMcpHandler` with the `/mcp` endpoint.
- **SSE (Server-Sent Events) transport is deprecated** — do not use for new builds.
- **10ms CPU limit on free tier is the real constraint.** A single MCP tool call that fetches data from multiple APIs, runs a cost basis calculation over 1,000 transactions, and formats the response will easily exceed 10ms of CPU. This forces the $5/month paid plan (50ms CPU) from day one of having real users.
- **Workaround for heavy computation:** Offload computation to D1 queries (database does the work, not the Worker CPU); cache intermediate results in KV.

## 5. Legal / Compliance Angle

### Thailand: Does a Crypto Tax Calculation Tool Require a License?

**Short answer: No.**

Thailand's licensing requirements under the 2018 Digital Asset Business Decree apply to operators of **exchanges, brokers, and dealers** — businesses that custody assets, execute trades, or deal in digital tokens on behalf of clients. The minimum capital requirement is THB 50M for exchanges, THB 10M for DEXes.

A tax calculation tool — software that reads transaction history and computes gain/loss — is not trading, brokering, or dealing. It is a **software product**, not a financial services business. No Thai SEC license is required.

The Thai SEC does regulate digital asset **advisors** (those giving investment advice on which tokens to buy/sell). BiG's MCP must explicitly avoid this: do not generate buy/sell recommendations; do not suggest tax strategies beyond factual calculation. Stick to computation: "Here is your FIFO cost basis and realized gain for these transactions." That is not investment advice.

[globallegalinsights.com/practice-areas/blockchain-cryptocurrency-laws-and-regulations/thailand] [silklegal.com/licenses-digital-assets-2] [bakermckenzie.com/en/insight/publications/guides/guide-to-cryptocurrency-in-thailand]

### US: Selling Crypto Tax Tools to US Customers as a Thai Business

**Key risks and mitigations:**

1. **Not a registered investment advisor (RIA) / no FinCEN registration required** — A pure tax calculation tool that does not offer investment advice, does not hold user funds, and does not execute transactions does not trigger RIA, MSB (Money Services Business), or FinCEN registration requirements. TurboTax, Koinly, and CoinLedger operate without these licenses.

2. **IRS preparer rules:** BiG's MCP is not a "paid tax preparer" — it provides data and calculations but does not sign or file returns on behalf of users. No PTIN (Preparer Tax Identification Number) required.

3. **Disclaimer is mandatory:** The product must clearly state "This tool provides calculations for informational purposes only and does not constitute tax or financial advice. Consult a qualified tax professional." This is standard practice for all crypto tax tools and provides meaningful legal protection.

4. **US consumer protection laws apply if targeting US customers:** Terms of Service must be clear, refund policy stated, and no deceptive marketing. Standard SaaS boilerplate covers this.

[sanctionscanner.com/blog/cryptocurrency-in-the-united-state-1141]

### GDPR / Privacy — Wallet Addresses and API Keys

**Wallet addresses are personal data under GDPR:**
- The EDPB (European Data Protection Board) issued guidelines in April 2025 treating blockchain data like any other technology — no special exemptions.
- A wallet address combined with transaction history can identify a real person via exchange KYC records or blockchain analytics. GDPR applies if any EU users are served.
- The final EDPB guidelines on blockchain and GDPR were expected by late 2025/early 2026.

**Practical implications for BiG:**

| Data | Status | Mitigation |
|------|--------|-----------|
| Wallet addresses | Personal data under GDPR | Store only what's needed; allow deletion |
| CEX API keys | Sensitive credentials | Encrypt at rest (AES-256); never log; allow user revocation |
| Transaction history | Personal financial data | Store in D1 scoped by user ID; delete on account close |
| Price data | Not personal | Cache freely |

**Thailand PDPA:** Thailand's Personal Data Protection Act mirrors GDPR in most obligations (consent, purpose limitation, data subject rights). If BiG processes Thai user data, PDPA applies. Since BiG is operating in Thailand, PDPA compliance is mandatory regardless of where users are.

**Right to erasure:** Users must be able to delete their account and all associated data. D1 allows a `DELETE WHERE user_id = ?` to wipe all transaction records. KV allows key deletion. Implement this from day one — it is a launch blocker for EU users under GDPR.

**So what?** No financial license is needed in Thailand or the US for a calculation tool. Add a one-sentence disclaimer on every output. Implement: (1) encrypted KV storage for API keys, (2) account deletion that purges all user data from D1 and KV, (3) PDPA/GDPR-compliant privacy policy. These are 1–2 day implementation tasks, not blockers.
